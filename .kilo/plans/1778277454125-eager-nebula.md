# Zenith Alerts Implementation Plan

## Overview
This plan outlines the implementation of the Zenith Alerts Discord bot as specified in the Software Requirements Specification v2.0 (zenith_rebuild_srs_v2.md). The implementation will be a clean rebuild following the exact specifications, excluding any FNF keys feature (which is not part of the SRS v2.0).

## Phase 1: Project Setup
1. Initialize TypeScript project
   - `npm init -y`
   - Install TypeScript: `npm install typescript --save-dev`
   - Create `tsconfig.json`
2. Install dependencies
   - Runtime: `discord.js@14`, `dotenv`, `prisma`, `@prisma/client`, `zod`
   - Dev: `@types/node`, `ts-node-dev` (or `ts-node` and `nodemon`), `@types/discord.js`
3. Set up Prisma
   - `npx prisma init`
   - Configure `datasource db` to use PostgreSQL via `DATABASE_URL`
4. Create directory structure as per SRS Section 12

## Phase 2: Core Infrastructure
1. Implement environment validation (`src/config/env.ts` using zod)
2. Create logger utility (`src/utils/logger.ts`) for structured JSON logging
3. Set up error handling (`src/utils/errors.ts`) with custom AppError class
4. Implement permission utilities (`src/utils/permissions.ts`) for admin/owner checks
5. Create Discord client setup (`src/bot/client.ts`)
6. Implement command deployment (`src/bot/registerCommands.ts`)
7. Create interaction router (`src/bot/interactionRouter.ts`)

## Phase 3: Encryption and Security
1. Implement encryption utilities (`src/modules/users/encryption.util.ts`)
   - AES-256-GCM encryption/decryption with unique IV per record
   - Key masking function
   - Environment key validation on startup
2. Ensure no secrets are logged (use masked keys only)

## Phase 4: User Registration System
1. Create UserRegistration and UserSettings Prisma models (already in schema.prisma)
2. Implement user repository (`src/modules/users/user.repository.ts`)
3. Implement user service (`src/modules/users/user.service.ts`)
4. Create register command (`src/commands/register.command.ts`)
   - `/register add <key>`: validate, encrypt, upsert, create default settings
   - `/register list`: show masked key and registration date
   - `/register remove`: confirmation button, delete user data
5. All registration commands must be ephemeral

## Phase 5: User Settings System
1. Implement settings command (`src/commands/settings.command.ts`)
   - Show current settings embed with "Edit" button
   - Handle modal submission for toggles (DD, Ping, Trench, Nuke, Broadcast Alerts)
   - Persist changes to UserSettings
   - Error handling for unregistered users
2. Ensure settings changes take effect immediately

## Phase 6: Pushover Integration
1. Create Pushover client (`src/modules/pushover/pushover.client.ts`)
   - HTTP wrapper for Pushover API
   - Handle different alert types (silent, bell, critical) with proper parameters
   - Implement error handling (400, 429, 500, timeouts) without crashing
   - Implement rate limiting (max 10 concurrent, batch delays)
2. Implement Pushover service (`src/modules/pushover/pushover.service.ts`)
   - Broadcast logic with filtering by user settings
   - Parallel sending with concurrency limit
   - Result aggregation (success/failure counts)

## Phase 7: Alert Broadcast Commands
1. Create alertAll command (`src/commands/alertAll.command.ts`)
   - Implement `/alert all silent <message>` (priority -1)
   - Implement `/alert all bell <message>` (priority 1, sound: pushover)
   - Implement `/alert all critical <message>` (priority 2, sound: siren, retry: 60, expire: 3600)
   - Permission check: admin role or ADMINISTRATOR permission
   - Filter recipients based on alert type and user settings
   - Log results to AlertAuditLog
   - Post history embed in configured channel
   - Reply with notification summary

## Phase 8: Alert History and Audit
1. Implement alert repository (`src/modules/alerts/alert.repository.ts`)
2. Implement alert service (`src/modules/alerts/alert.service.ts`) for logging
3. Create alert history embed format as specified (color coding, jump link)
4. Ensure history persists across bot restarts

## Phase 9: Admin Tools
1. Create admin command (`src/commands/admin.command.ts`)
   - `/admin config`: show current settings, buttons to set history channel and manage admin roles
   - `/admin health`: check database, Pushover API, Discord gateway, uptime, registered users
2. Implement guild repository and service (`src/modules/guilds/*.ts`)
3. Persist guild configuration (alert history channel ID, admin role IDs)

## Phase 10: Testing
1. Write unit tests for:
   - Encryption/decryption (`tests/unit/encryption.test.ts`)
   - Key masking (`tests/unit/masking.test.ts`)
   - Permission checks (`tests/unit/permissions.test.ts`)
2. Write integration tests for:
   - Registration flow (`tests/integration/register.test.ts`)
   - Broadcast flow (`tests/integration/broadcast.test.ts`)
3. Aim for minimum coverage as specified in SRS Section 15.3

## Phase 11: Deployment
1. Create Dockerfile as per SRS Section 16.1
2. Create docker-compose.yml with postgres and bot services
3. Ensure Prisma migrations run on container startup
4. Provide .env.example with required variables

## Phase 12: Documentation
1. Create README.md with setup instructions
2. Ensure .env.example is present
3. Copy SRS v2.0 to docs/SRS.md
4. Add comments to complex functions as needed

## Exclusions (FNF Keys Feature)
As per user request, the FNF keys feature (not present in SRS v2.0) will not be implemented. Any existing FNF-related code in the current repository will be ignored since this is a clean rebuild. If working in the existing directory, we will focus only on the new structure and overwrite/remove conflicting files.

## Implementation Order
Following the SRS Section 15.5 Build Order:
1. Init TypeScript project with discord.js, Prisma, dotenv
2. Create Prisma schema
3. Implement env validation (use zod)
4. Setup Discord client and command registration
5. Implement `/help` command
6. Implement `/register` commands (add/list/remove)
7. Implement encryption utils (encrypt/decrypt/mask)
8. Implement `/settings` command with modal
9. Implement Pushover client (HTTP wrapper)
10. Implement `/alert all` commands (silent/bell/critical)
11. Implement alert history embed posting
12. Implement `/admin` commands
13. Add tests
14. Add Dockerfile + docker-compose.yml
15. Write README with setup instructions

## Quality Assurance
- Ensure TypeScript compiles with no errors
- Never log raw Pushover keys, encryption keys, or secrets
- Commands should be thin (parse input, call services)
- Business logic in services, database access in repositories, HTTP calls in clients
- Structured JSON logging for important events
- Fail fast on missing required environment variables
